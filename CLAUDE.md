# HivePulse — Claude Instructions

## Project Overview

HivePulse is an open-source uptime monitoring and status page platform.
Target users: infrastructure engineers, homelab operators, SaaS teams (5–500 endpoints).
License: MIT core, optional BSL premium modules.

## Repository Structure

```
hivepulse/
  hivepulse-api/    # Go backend
  hivepulse-web/    # React + TypeScript frontend
```

## Backend (Go)

### Tech Stack
- Language: Go
- Framework: Gin (HTTP), GORM (ORM)
- Database: PostgreSQL 16
- Auth: JWT (access token in body, refresh token as httpOnly cookie)
- API Docs: swaggo/swag (Swagger 2.0 / OpenAPI)
- Migrations: golang-migrate
- Scheduler: custom goroutine-based
- WebSocket: hub pattern in `internal/infrastructure/websocket_hub.go`

### Architecture: Clean Architecture (Hexagonal)
Dependencies always point inward. Never import outer layers from inner layers.

```
domain/        — pure Go structs & domain errors, zero dependencies
port/          — interfaces (repository, service/checker/notifier)
usecase/       — business logic, depends only on domain + port interfaces
adapter/
  handler/     — Gin HTTP handlers
  repo/        — GORM repository implementations
  service/     — external integrations (checkers, notification senders)
  middleware/  — JWT, rate limiter, CORS, logger, role guard
infrastructure/ — DB connection, config, scheduler, WebSocket hub
```

### Directory Layout
```
hivepulse-api/
  cmd/server/main.go
  internal/domain/
  internal/port/
  internal/usecase/
  internal/adapter/handler/
  internal/adapter/repo/
  internal/adapter/service/
  internal/adapter/middleware/
  internal/infrastructure/
  docs/                        # auto-generated by swag, never edit manually
  migrations/
  Dockerfile
  docker-compose.yml
  Makefile
  .env.example
```

### Coding Rules — Backend
- All business logic lives in `usecase/`. Handlers are thin.
- Handlers never import `repo/` or `service/` directly — only via `port/` interfaces.
- GORM models live in `adapter/repo/`, not in `domain/`.
- Domain structs are plain Go structs with no GORM/Gin tags.
- Every handler must have Swagger annotations: `@Summary`, `@Tags`, `@Param`, `@Success`, `@Failure`, `@Router`, `@Security`.
- Run `swag init --generalInfo cmd/server/main.go --output docs/` after any handler change.
- Never hard-code secrets; always load from environment variables.
- All JWT secrets, DB credentials, and SMTP passwords come from env vars.
- Use `bcrypt` cost factor 12 for password hashing.
- Refresh token rotation: every use invalidates the old token and issues a new pair.
- Rate limit `/auth/login`: 5 attempts/min/IP.

### Monitoring Engine
- Supported check types: HTTP/HTTPS, TCP, PING (ICMP), DNS, SMTP
- Intervals: 10s, 30s, 60s, 120s, 300s, 600s, 1800s, 3600s
- Retries: 0–5, configurable retry interval (default 20s)
- Request timeout: 5–120s (default 30s)
- SSL cert expiry warnings at: 7, 14, 30, 60 days

### Data Retention (Aggregator)
Aggregator goroutine runs every 5 minutes — idempotent, safe to interrupt.

| Tier          | Retention     |
|---------------|---------------|
| Raw heartbeats| 48 hours      |
| stats_minutely| 7 days        |
| stats_hourly  | 90 days       |
| stats_daily   | Unlimited (configurable via `HIVEPULSE_RETENTION_DAILY_DAYS`) |

### Database
- PostgreSQL 16 only (no SQLite, no MariaDB)
- `heartbeats` table: PARTITION BY RANGE (checked_at), daily partitions
- Key indexes: `(monitor_id, checked_at DESC)` on heartbeats, `(monitor_id, bucket DESC)` on stats_daily
- JSONB with GIN index for `notification_channels.config`
- PgBouncer (transaction mode) for connection pooling in production

## Frontend (React)

### Tech Stack
- Framework: React + TypeScript
- Build: Vite
- Styling: Tailwind CSS
- State: Zustand (auth/token state only, in memory — never localStorage)
- Data fetching: React Query (TanStack Query)
- HTTP: Axios with JWT interceptor + silent refresh on 401
- WebSocket: custom `wsClient.ts`
- Fonts: Bricolage Grotesque (headings/stats/wordmark) + IBM Plex Mono (data/URLs/timestamps)

### Architecture: Clean Architecture
```
src/domain/           — TypeScript interfaces: Monitor, User, Incident, Notification, CheckResult
src/application/      — React Query hooks: useMonitors, useAuth, useNotifications, etc.
src/infrastructure/   — apiClient.ts, wsClient.ts
src/presentation/
  pages/              — DashboardPage, LoginPage, RegisterPage, StatusPage, SettingsPage
  components/         — MonitorCard, HexStatus, Sidebar, Heatmap, Toast, MonitorForm, NotificationForm
src/shared/           — ThemeProvider, useTheme, utility functions, constants
```

### Coding Rules — Frontend
- Access token stored in Zustand memory store only — never in localStorage or sessionStorage.
- Axios request interceptor attaches Bearer token to every call.
- Axios response interceptor: on 401, call `/auth/refresh`, retry original request, redirect to login only if refresh fails.
- All API types defined in `src/domain/` as TypeScript interfaces.
- Data fetching logic lives in `src/application/` hooks, not in components.
- Components in `src/presentation/` must not import from `src/infrastructure/` directly.

### Brand & UI
- Dark mode background: `#202232` (navy-black)
- Light mode background: `#F5F3EC` (warm cream)
- Theme persisted in user preference, toggled via `useTheme` hook
- Signature component: `HexStatus` — animated SVG hexagon, stroke fills proportionally to uptime %, pulsing heartbeat polyline inside
- Honeycomb SVG background pattern at very low opacity
- Toast notifications: bottom-right, color-coded (red=critical, amber=warning, green=recovery), auto-dismiss 4s

## Authentication & RBAC

### JWT Flow
- POST `/auth/login` → access token (response body) + refresh token (httpOnly cookie)
- Access token: short-lived, sent as `Authorization: Bearer <token>` on every request
- Refresh token: long-lived, stored in PostgreSQL as SHA-256 hash with device fingerprint, IP, expiry
- POST `/auth/refresh` → new access token + rotated refresh token

### Roles
- **Admin**: full access
- **Editor**: manage monitors, incidents, notifications; no user management
- **Viewer**: read-only dashboard and status page

## Testing Strategy

### Backend (Go)
- Unit tests: use case layer with mock repositories (testify/mock or gomock). Target: 80%+ coverage on usecases.
- Integration tests: `testcontainers-go` with real PostgreSQL — no mocking the database.
- API tests: `httptest` + Gin test mode.
- Linter: `golangci-lint` strict preset (errcheck, gosec, govet, staticcheck).

### Frontend (React)
- Unit tests: Vitest + React Testing Library. Target: 70%+ coverage on `application/` and `presentation/`.
- Integration tests: MSW (Mock Service Worker) for API mocking.
- E2E (Phase 4): Playwright for critical user journeys.

### CI (GitHub Actions)

#### PR Checks (every pull request)
- Go lint (`golangci-lint`) + test + build
- Frontend lint + test + build
- Swagger diff check (`swag init && diff docs/`)

#### Merge to `main`
- Docker image build + push to GHCR (GitHub Container Registry) — internal/dev channel
- Semantic versioning via conventional commits

#### Release Pipeline (on git tag `v*.*.*`)
Triggered by pushing a semver tag (e.g. `git tag v1.2.0 && git push --tags`).

**Docker Hub**
- Build multi-arch image (linux/amd64, linux/arm64) using `docker buildx`
- Push to `hivepulse/hivepulse-api` and `hivepulse/hivepulse-web` on Docker Hub
- Tag strategy: `v1.2.0`, `v1.2`, `v1`, `latest`
- Credentials: `DOCKERHUB_USERNAME` + `DOCKERHUB_TOKEN` secrets in GitHub Actions

**Helm Chart**
- Chart lives in `charts/hivepulse/` within the repo
- On release: package chart (`helm package`), update `index.yaml`, push to GitHub Pages branch (`gh-pages`) as a Helm repository
- Chart version bumped to match app version automatically
- Users install via: `helm repo add hivepulse https://hivepulse.github.io/hivepulse && helm install hivepulse hivepulse/hivepulse`

**OpenQL**
- Release artifacts published to OpenQL registry as part of the tag pipeline
- Credentials managed via `OPENQL_TOKEN` secret in GitHub Actions

**GitHub Release**
- Automatically created via `gh release create` or `softprops/action-gh-release`
- Attached artifacts: `docker-compose.yml`, `.env.example`, `charts/` tarball, checksums (`SHA256SUMS`)
- Release notes auto-generated from conventional commit messages since last tag

#### Versioning Convention
- Use [Conventional Commits](https://www.conventionalcommits.org/): `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`, `test:`
- `feat` → minor bump, `fix` → patch bump, `feat!` or `BREAKING CHANGE` → major bump
- Version source of truth: git tag; Go binary embeds version via `-ldflags "-X main.version=$(git describe --tags)"`

## Deployment

### Docker Compose (default)
- `hivepulse-api`: multi-stage Dockerfile (build: `golang:1.26-alpine`, run: `alpine:3.19`)
- `postgres`: `postgres:16-alpine` with named volume
- Optional: PgBouncer, Redis (for distributed rate limiting)

### Key Environment Variables
- `HIVEPULSE_RETENTION_DAILY_DAYS` — stats_daily retention (default: unlimited)
- `JWT_ACCESS_SECRET`, `JWT_REFRESH_SECRET` — never hard-code
- All secrets via env vars only

## What to Avoid
- Do not import GORM or Gin packages inside `domain/` or `usecase/`.
- Do not store tokens in localStorage or sessionStorage.
- Do not hard-code any secret or credential.
- Do not edit files inside `docs/` manually — always regenerate with `swag init`.
- Do not use SQLite or any database other than PostgreSQL.
- Do not mock the database in integration tests — use `testcontainers-go` with real PostgreSQL.
- Do not skip Swagger annotations on new handlers.
