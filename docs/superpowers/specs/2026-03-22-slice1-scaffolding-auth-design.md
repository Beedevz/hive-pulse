## 1. Context & Goals

HivePulse is an open-source uptime monitoring and status page platform. Development follows a **Full-Stack Slice** strategy: each slice delivers a working, testable vertical cut of the product before moving to the next.

Slice 1 is the foundation. When complete:
- The monorepo is scaffolded with Clean Architecture on both backend and frontend
- A developer can `docker compose up` and get a running app
- A one-time setup flow creates the first admin
- JWT-based auth (login, refresh, logout, me) works end-to-end
- CI runs lint + test + build on every PR

## 4. Auth Design

### One-Time Setup Endpoint

| Endpoint | Method | Auth | Description |
|----------|--------|------|-------------|
| `/auth/setup` | POST | None | Creates first admin. Returns 403 if users table is non-empty. |

**Middleware guard:** Before handling, query `SELECT COUNT(*) FROM users`. If > 0, return `403 Forbidden` with `{"error": "setup already completed"}`.

**Frontend flow:**
1. App starts → calls `GET /api/v1/auth/setup/status`
2. If `setup_required: true` → redirect to `/setup`
3. If `setup_required: false` → call `GET /api/v1/auth/me` (with stored access token)
4. If `/auth/me` returns 401 → redirect to `/login`

### Setup Status Endpoint

| Endpoint | Method | Auth | Description |
|----------|--------|------|-------------|
| `/auth/setup/status` | GET | None | Returns `{"setup_required": true}` if users table is empty, `{"setup_required": false}` otherwise. Always public. Full URL: `GET /api/v1/auth/setup/status`. Used by frontend on app start to decide routing. |

### JWT Auth Endpoints

| Endpoint | Method | Auth | Description |
|----------|--------|------|-------------|
| `/auth/login` | POST | None | email+password → access token (body) + refresh token (httpOnly cookie) |
| `/auth/refresh` | POST | Cookie | Rotates refresh token, returns new access token |
| `/auth/logout` | POST | Bearer | Deletes refresh token from DB, clears cookie |
| `/auth/me` | GET | Bearer | Returns current user info |

**Token architecture:**
- Access token: 15-minute expiry, returned in response body
- Refresh token: 7-day expiry, stored as SHA-256 hash in DB, sent as `httpOnly; Secure; SameSite=Strict` cookie
- Refresh token rotation: each use invalidates previous, issues new pair

**Security:**
- Passwords: bcrypt cost factor 12
- Rate limiting: 5 attempts/min/IP on `/auth/login`
- CORS: explicit origin whitelist, `credentials: true`
- All secrets via env vars only

**Frontend token management:**
- Access token → Zustand memory store (never localStorage/sessionStorage)
- Axios request interceptor: attach `Authorization: Bearer <token>`
- Axios response interceptor: on 401 → call `/auth/refresh` → retry original → on failure → redirect to `/login`

## 5. Database Schema

### Migration: `000001_init_users.up.sql`

```sql
CREATE TABLE users (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email       TEXT NOT NULL UNIQUE,
    name        TEXT NOT NULL,
    password_hash TEXT NOT NULL,
    role        TEXT NOT NULL DEFAULT 'viewer' CHECK (role IN ('admin','editor','viewer')),
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE refresh_tokens (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    token_hash  TEXT NOT NULL UNIQUE,
    device_fp   TEXT,   -- stored but not enforced in Slice 1; reserved for future device-aware revocation
    ip          TEXT,   -- stored for audit purposes only in Slice 1
    expires_at  TIMESTAMPTZ NOT NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_refresh_tokens_user_id ON refresh_tokens(user_id);
CREATE INDEX idx_refresh_tokens_expires_at ON refresh_tokens(expires_at);
```

## 7. Infrastructure

### Docker Compose

In Slice 1, Docker Compose runs only the backend (`api`) and database (`postgres`). The frontend runs separately via `npm run dev` (Vite dev server on `:5173`). A frontend Docker service will be added in a later slice when the product nears a release.

```yaml
services:
  api:
    build: ./hivepulse-api
    env_file: .env
    ports: ["8080:8080"]
    depends_on:
      postgres: { condition: service_healthy }

  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: hivepulse
      POSTGRES_USER: hivepulse
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
    volumes: [postgres_data:/var/lib/postgresql/data]
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U hivepulse"]
      interval: 5s
      timeout: 5s
      retries: 5

volumes:
  postgres_data:
```

## 8. Testing Strategy (Slice 1)

### Backend
- **Unit tests:** `auth_usecase_test.go` — mock `UserRepository` and `RefreshTokenRepository` via `testify/mock`. Cover: setup guard, login success/failure, token refresh, logout.
- **Integration tests:** `testcontainers-go` spins up real PostgreSQL. Tests `user_repo.go` and `token_repo.go` against real DB.
- **API tests:** `httptest` + Gin test mode for auth handler endpoints.
- Target: 80%+ coverage on `usecase/`.

### Frontend
- **Unit tests:** Vitest + React Testing Library for `LoginPage`, `SetupPage`, `useAuth` hook.
- **Mock:** MSW to intercept `/auth/login`, `/auth/setup`, `/auth/me`, `/auth/refresh`.
- **Interceptor test:** Vitest test for `apiClient.ts` — simulate a 401 response, verify that the Axios interceptor calls `/auth/refresh` and retries the original request transparently.
- Target: 70%+ coverage on `application/` and `presentation/`.
